# 議論ループ・プロトコル

1トピックを前進させる標準サイクル。Orchestrator がこの順で各エージェントを「呼び出す」
(= 各ロール定義ファイルを読み、そのペルソナで独立に思考させる。サブエージェント実行環境があれば Task として分離実行する)。

## フェーズ

### 0. Framing(司令塔)
- トピックを1文で定義し、成功条件と制約を明示
- L0/L1 から関連文脈を要約して「共有ブリーフ」を作る(300字以内)
- 関与すべきロールを選定(全員招集はコスト高。既定は3〜5名)

### 1. Propose(起案)
- 通常 PdM が起案。技術トピックなら Engineer、体験トピックなら UIUX が起案
- 出力形式: 「提案 / 根拠 / 想定リスク / 代替案を捨てた理由」

### 2. Review(多職種レビュー)— 並列
- 各ロールは **自分の観点からのみ** 200〜400字でレビュー
- 必須出力: `[賛成/条件付き賛成/反対] + 理由 + 修正提案1つ以上`
- 同意だけのレビューは禁止。必ず1点は改善か懸念を出す

### 3. Challenge(反証)
- Consultant が悪魔の代弁者として最も強い反論を構築
- QA が「壊れ方」の観点で失敗シナリオを列挙

### 4. Decide(裁定)
- PM が全見解を踏まえ裁定。形式: 「決定 / 採用した理由 / 却下した意見とその理由 / 見直し条件」
- ラウンド上限3。3回で収束しなければ PM が仮決定(revisit条件付き)して前進

### 5. Record(記録)
- 議論全文 → `memory/L2-episodic/` に追記
- 決定 → `decisions/ADR-XXXX.md` + `L0-core/decisions.md` に索引1行
- 未解決事項 → `L1-working/open-questions.md`

### 6. Execute(実装)
- 作業ブランチ `feat/<topic>` を切る(モノレポ内、ADR-0001)
- Engineer が実装、UIUX が画面系を並走

### 7. Verify(検証)
- QA が受入条件に対してレビュー・テスト。不合格なら 6 に差し戻し(最大2回、超えたら 4 へ戻す)

### 8. Distill & Checkpoint
- 得られた知見を L3 へ蒸留(`scripts/distill.sh` 手順)
- マイルストーン/MVP 到達なら `scripts/checkpoint.sh` でコミット・main へマージ提案

## 人間への確認が必要になったとき
即質問は禁止。以下のフォーマットで提示する:

```
## 判断依頼: <トピック>
検討状況: <どこまで議論し、何が対立点か 3行以内>
選択肢A: <内容 / 支持ロール / メリット / リスク>
選択肢B: <同上>
PM推奨: <AかB + 理由>
デフォルト動作: 24h相当の応答がなければ推奨案で前進します
```
